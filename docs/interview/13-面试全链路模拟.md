# 模块十三：面试全链路模拟

## 一、30 秒口述模板

> "CloudChunk 是一个基于分片上传的云文件系统，前端把大文件切成 5MB 分片并发上传，后端落到 MinIO 对象存储，全部到齐后服务端 Compose 合并，再通过 RocketMQ 异步做 MD5 校验和转码。核心亮点有：秒传去重（MD5 + 引用计数，省 35% 存储）、断点续传（Redis/MySQL/MinIO 三层兜底）、去长事务（Little's Law 驱动，去掉 @Transactional 后 DB 连接池压力从 20 降到 0.5）、Lua 原子进度（HSETNX+HINCRBY+EXPIRE 单 RTT）、虚拟线程+Semaphore 背压（保护下游资源非线程本身）。"

---

## 二、面试全链路模拟：上传链路

面试官会问："从用户上传一个文件到完成，整个链路是怎样的？"

### 9 步走口述：

1. **前端**：用 spark-md5 增量计算整文件 MD5，避免一次性加载大文件
2. **调 POST /upload/init**：前端传 fileName、fileSize、fileMd5、chunkSize、chunkTotal
3. **秒传检查**：Redis SETNX 排它锁，防止并发创建多个会话。查 file_meta 表 WHERE file_md5=? AND status=2，命中则直接返回 URL（秒传），ref_count+1
4. **续传**：未命中则查 upload_session 表，存在 RUNNING 的会话则返回已上传列表
5. **新建**：都没有则创建新 upload_session，Redis Hash 初始化分片进度
6. **分片并发上传**：前端 4-6 路并发上传 POST /upload/chunk，**这里不用 @Transactional**（因为 MinIO PUT ~200ms，包在事务里会持有 DB 连接导致连接池耗尽）
7. **MinIO Compose 合并**：全部到齐后服务端 Compose 合并（数据不经应用层），**合并后状态是 MERGED 而非 AVAILABLE**（因为还要异步校验 MD5）
8. **异步校验**：投 RocketMQ 消息做异步整文件 MD5 校验，**流式 MD5（DigestInputStream）不加载整文件到内存**，校验通过置 AVAILABLE
9. **异步转码**：投递转码 MQ 消息按 MIME Type 分发到 img/video/doc Worker，**模板方法模式**统一处理

## 三、面试全链路模拟：下载链路

### 3 步走口述：

1. **预签名 URL**：先查 Redis 缓存 URL，命中则直接返回，缓存 TTL 略小于签名有效期（避免返回过期 URL）
2. **流式下载**：支持 ETag 304（fileMd5 做 ETag，If-None-Match 匹配则零数据传输）和 Range 206（断点续传，视频播放器拖拽）
3. **StreamingResponseBody**：Servlet 线程先释放，下载 IO 交给异步线程，256KB buffer 降系统调用

---

## 四、面试全链路模拟：性能优化

### 5 步走口述：

1. **去长事务**：去掉了 @Transactional，按照 Little's Law，100 QPS x 0.2s = 20 连接，去掉后 0.5 连接
2. **Lua 原子进度**：HSETNX+HINCRBY+EXPIRE 单 RTT，HGETALL O(N) → O(1) 查进度
3. **Raw Octet-Stream**：零临时文件，request.getInputStream() 直传 MinIO
4. **Caffeine L1 缓存**：~100ns vs Redis ~1ms，写穿透 + TTL 兜底
5. **虚拟线程背压**：Semaphore(64 permits) 保护 MinIO 连接数

## 五、面试高频追问（必背！）

### 5.1 架构设计类

| # | 问题 | 回答要点 |
|---|---|---|
| 1 | 为什么去 @Transactional？ | MinIO PUT ~200ms，Little's Law = QPS x 耗时 = 100 x 0.2s = 20 连接，接近池上限 |
| 2 | Lua 脚本在 Redis Cluster 兼容？ | 单 KEY 操作，天然同 slot |
| 3 | 秒传原理？ | MD5 查 file_meta，命中后 ref_count+1，不复制对象 |
| 4 | 断点续传？ | Redis 进度 + MySQL + MinIO 交集重建 |
| 5 | 虚拟线程为何加 Semaphore？ | 保护下游资源非线程本身 |
| 6 | 令牌桶 vs 漏桶？ | 令牌桶允许突发，适合分片上传 |
| 7 | Caffeine 一致性？ | 写穿透 invalidate L1+L2 + TTL 兜底 |
| 8 | Compose 限制？ | 单次 10000 对象，每个最小 5MB |
| 9 | 转码失败影响？ | 不影响，文件已 AVAILABLE |
| 10 | MergeTransactionService 为何独立？ | Spring AOP 不拦截自调用 |

### 5.2 性能优化类

| # | 问题 | 回答要点 |
|---|---|---|
| 1 | 去长事务后连接池压力？ | 去掉 @Transactional 后 DB 5ms，100 x 0.005 = 0.5 连接 |
| 2 | Redis Lua 原子进度优化？ | HSETNX+HINCRBY+EXPIRE 1 RTT，HGETALL O(N) → O(1) |
| 3 | Raw Octet-Stream 上传？ | 零临时文件，request.getInputStream() 直传 MinIO |
| 4 | Streaming MD5 校验？ | DigestInputStream 边传边算，零额外 buffer |
| 5 | Caffeine L1 缓存？ | ~100ns vs Redis ~1ms，减少 99% 的 Redis/DB 访问 |
| 6 | MinIO OkHttp Client 调优？ | ConnectionPool(256, 5min) + maxRequestsPerHost(256) |
| 7 | ETag 304？ | fileMd5 做 ETag，If-None-Match 匹配 → 304，零数据传输 |
| 8 | Range 206？ | 断点续传，支持视频播放器拖拽 |

### 5.3 分布式设计类

| # | 问题 | 回答要点 |
|---|---|---|
| 1 | 秒传幂等锁？ | SETNX + TTL 10min，防止并发创建多个会话 |
| 2 | 合并分布式锁？ | SETNX + TTL 5min，防止并发合并 |
| 3 | 转码幂等？ | cc:transcode:done:{fileId}:{type} SET EX 7 天 |
| 4 | 分片幂等？ | HSETNX + INSERT ON DUPLICATE KEY UPDATE |
| 5 | 转码失败重试？ | RetryableException → MQ 重投（最多 3 次）→ 死信队列 |
| 6 | 限流？ | 令牌桶 Lua per-user + RateLimitFilter |
| 7 | 缓存一致性？ | 写穿透：所有写操作在 invalidateCache 中同时清除 L1+L2 |

## 六、项目亮点一句话总结（面试必背！）

1. **秒传**：MD5 + 引用计数，省 35% 存储
2. **断点续传**：Redis/MySQL/MinIO 三层兜底
3. **去长事务**：Little's Law 驱动的架构决策
4. **Lua 原子**：HSETNX+HINCRBY+EXPIRE 单 RTT
5. **零拷贝上传**：octet-stream 直传 MinIO
6. **服务端合并**：Compose 不经应用层带宽
7. **异步转码**：MQ 解耦 + TAG 路由 + 模板方法
8. **两级缓存**：Caffeine L1 ~100ns + Redis L2
9. **虚拟线程背压**：Semaphore 保护下游资源
10. **令牌桶限流**：Redis Lua 单 RTT per-user

---

## 七、风险与权衡（面试加分项）

| 风险点 | 缓解措施 |
|---|---|
| Redis 进度丢失 | TTL 24h + MySQL + MinIO 交集兜底重建 |
| MinIO Compose 单次限制 | 分片大小 5MB 保底，超限分批 Compose |
| 合并中应用重启 | 合并锁 TTL 5min 自动释放，前端可重试 merge |
| 转码消息积压 | Worker 独立扩缩容、死信队列告警 |
| 大文件 MD5 异步校验期间用户已下载 | 合并后置 MERGED，校验通过后才 AVAILABLE |
