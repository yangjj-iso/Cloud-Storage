# 模块八：下载链路

---

## 一、下载链路概览

用户下载文件有两种方式，各走不同的路径：

| 方式 | 谁传数据 | 适用场景 |
|---|---|---|
| **预签名 URL** | MinIO 直接传，应用层只做签名 | 大文件下载，减轻应用层压力 |
| **代理下载** | 应用层流式返回 | 需要鉴权 / 需要断点续传 / 需要缓存协商 |

> 🎯 通俗理解：**预签名 URL** 就像你拿了一张盖了章的门票，自己去 MinIO 那取东西；**代理下载** 就像你让快递员帮你取，快递员帮你把东西送到你手上。

---

## 二、预签名 URL + 缓存

### 2.1 什么是预签名 URL？

MinIO 的预签名 URL 是一种"临时通行证"，里面包含了：
- 文件位置（bucket + objectKey）
- 过期时间（TTL）
- 签名（防篡改）

用户拿到这个 URL 后，**直接从 MinIO 下载**，不经过应用层，所以应用层压力很小。

### 2.2 核心代码

```java
// DownloadService.java
public PresignedUrl presign(String fileId, Duration ttl) {
    // 1. 先查文件元数据，确保文件存在且可用
    FileMeta meta = fileMetaService.getAvailableOrThrow(fileId);

    // 2. 查 Redis 缓存，看有没有之前生成过的预签名 URL
    String cached = fileMetaService.getCachedUrl(fileId);
    if (cached != null && !cached.isBlank()) {
        return new PresignedUrl(cached, fileMetaService.getCachedUrlTtl(fileId));
    }

    // 3. 缓存没命中，调 MinIO 生成预签名 URL
    String url = storageFactory.current().presignDownload(meta.getBucket(), meta.getObjectKey(), ttl);

    // 4. 缓存 TTL 略小于签名有效期（避免返回已过期的 URL）
    Duration cacheTtl = ttl.compareTo(Duration.ofMinutes(5)) > 0 ? ttl.minusMinutes(1) : ttl;
    fileMetaService.cacheUrl(fileId, url, cacheTtl);

    return new PresignedUrl(url, cacheTtl);
}
```

### 2.3 用大白话解释流程

1. **先查 Redis 缓存**：之前已经生成过预签名 URL 且没过期？直接返回，省得再调 MinIO
2. **未命中：调 MinIO 签名**：让 MinIO 生成一个带临时签名的 URL
3. **缓存 TTL 略小于签名有效期**：比如签名有效期 5 分钟，缓存 TTL 设为 4 分钟。这样用户拿到的 URL 不会在缓存里躺着过期了，结果用的时候发现签名已经失效了

> 🎯 为什么缓存 TTL 要小于签名有效期？**举个例子**：签名的有效期是 5 分钟，你把缓存 TTL 设成 5 分钟。那在第 5 分钟的最后一秒，缓存里还存着 URL，但用户拿去用的时候签名已经过期了！所以缓存 TTL 要比签名有效期短一点，比如少 1 分钟，留出余量。

## 三、流式下载 + ETag 304 + Range 206

代理下载（`/api/files/{fileId}/download`）是应用层流式返回，支持三个 HTTP 特性：

| 特性 | HTTP 状态码 | 干什么用的 |
|---|---|---|
| **ETag 304** | 304 Not Modified | 文件没变化就不传数据，浏览器用缓存 |
| **Range 206** | 206 Partial Content | 断点续传，视频播放器拖拽进度条 |
| **StreamingResponseBody** | — | 不阻塞 Servlet 线程，异步 IO 下载 |

### 3.1 核心代码

```java
// FileController.download()
@GetMapping("/{fileId}/download")
public ResponseEntity<StreamingResponseBody> download(
        @PathVariable String fileId,
        @RequestHeader(value = HttpHeaders.RANGE, required = false) String range,
        @RequestHeader(value = HttpHeaders.IF_NONE_MATCH, required = false) String ifNoneMatch) {

    // 1. 查文件元数据，确保文件存在且可用
    FileMeta meta = fileMetaService.getAvailableOrThrow(fileId);

    // 2. ETag 304：判断文件是否变化，没变化就 304
    String etag = "\"" + meta.getFileMd5() + "\"";
    if (etag.equals(ifNoneMatch)) {
        return ResponseEntity.status(HttpStatus.NOT_MODIFIED).eTag(etag).build();
    }

    // 3. 打开下载流（支持 Range 206）
    DownloadService.DownloadStream ds = downloadService.open(fileId, range);
    HttpStatus status = ds.full() ? HttpStatus.OK : HttpStatus.PARTIAL_CONTENT;

    // 4. 用 StreamingResponseBody 流式返回，不阻塞 Servlet 线程
    StreamingResponseBody body = out -> {
        try (InputStream in = ds.stream()) {
            byte[] buf = new byte[256 * 1024]; // 256KB 缓冲区
            int n;
            while ((n = in.read(buf)) > 0) out.write(buf, 0, n);
            out.flush();
        }
    };

    return new ResponseEntity<>(body, headers, status);
}
```

### 3.2 ETag 304 —— 用大白话解释

> 🎯 **ETag 就是文件的"身份证号"。** ETag = fileMd5，文件内容不变，MD5 不变，天然 ETag。

**场景：** 用户第二次下载同一个文件

- **第一次请求**：服务器返回 200 + 文件内容 + 响应头 `ETag: "abc123"`
- **第二次请求**：浏览器自动带上 `If-None-Match: "abc123"`
- **服务器比对**：ETag = fileMd5，如果文件没变化，MD5 没变，ETag 没变 → 返回 **304 Not Modified**，零数据传输
- **浏览器收到 304**：直接用本地缓存，不用传数据

> 🎯 **好处：** 用户第二次下载同一个文件，浏览器直接用缓存，不用传数据，省带宽省时间。

### 3.3 Range 206 —— 用大白话解释

> 🎯 **Range 就是"断点续传"，视频播放器拖拽进度条就是用这个。**

**场景：** 用户看视频，拖到进度条中间位置

- **请求头**：`Range: bytes=0-1048575`（请求前 1MB）
- **响应**：`206 Partial Content` + `Content-Range: bytes 0-1048575/1073741824`
- **服务器只返回请求的那部分数据**，不传整个文件

**三种 Range 格式：**

| 格式 | 含义 | 举例 |
|---|---|---|
| `bytes=0-1023` | 从第 0 字节到第 1023 字节 | 下载前 1KB |
| `bytes=500-` | 从第 500 字节到最后 | 下载后半部分 |
| `bytes=-500` | 最后 500 字节 | 下载尾部 |

### 3.4 StreamingResponseBody —— 用大白话解释

> 🎯 **普通 Servlet 处理：** 一个请求进来，Tomcat 分配一个线程，线程一直等到下载结束才释放。**如果下载 1 个大文件，这个线程就占着 1 个大文件下载的时间。** 下载 1 个 1G 的文件，线程可能占几十分钟。**Tomcat 默认线程数有限（200 个），** 下载 200 个大文件，线程就满了，**其他请求全部被阻塞。**

**StreamingResponseBody 好处：**

- **Servlet 线程先释放**：下载 IO 交给异步线程，Tomcat 的 Servlet 线程不阻塞
- **256KB buffer 降系统调用次数**：每次读 256KB，而不是每次读 1 字节，减少系统调用次数

> 🎯 **为什么是 256KB？** 256KB 是一个平衡点，既不会太大（占内存），也不会太小（系统调用太多）。一般 4KB~1MB 的 buffer 都可以，256KB 是经验值。

## 四、Range 请求头解析

RangeSpec 解析代码（面试要会讲）：

```java
public static RangeSpec parse(String header, long total) {
    if (header == null || header.isBlank()) return full(total); // 没有 Range，返回整个文件
    if (!header.startsWith("bytes=")) return invalid;           // 格式不对，返回无效

    // 解析三种格式：
    // bytes=0-1023    → start=0, end=1023（从第 0 字节到第 1023 字节）
    // bytes=500-      → start=500, end=total-1（从第 500 字节到最后）
    // bytes=-500      → start=total-500, end=total-1（最后 500 字节）

    // 非法 Range（比如 start > total） → 416 Range Not Satisfiable
}
```

**面试要会讲的点：**

| Range 格式 | 解析结果 | 示例（假设 total=1073741824） |
|---|---|---|
| `bytes=0-1023` | start=0, end=1023 | 前 1KB |
| `bytes=500-` | start=500, end=1073741823 | 从第 500 字节到最后 |
| `bytes=-500` | start=1073741324, end=1073741823 | 最后 500 字节 |
| 非法（如 start > total） | 返回 416 | **416 Range Not Satisfiable + Content-Range: bytes \*/total** |

> 🎯 **面试追问**：非法 Range 返回什么？**416 Range Not Satisfiable + Content-Range: bytes \*/total**，告诉客户端文件总大小，让客户端修正请求。

## 五、面试追问

| 问题 | 回答要点 |
|---|---|
| 为什么用 fileMd5 做 ETag？ | 文件内容不变 MD5 不变，天然 ETag，不需要额外计算 |
| 预签名 URL 缓存 TTL 略小于？ | 避免返回已过期 URL，留出余量 |
| StreamingResponseBody 好处？ | Servlet 线程先释放，下载 IO 交给异步线程，不阻塞 Tomcat 线程池 |
| 256KB 缓冲区？ | 减少 read/write 系统调用次数，平衡内存和性能 |
| Range 206 原理？ | 请求头 `Range: bytes=0-1048575`，响应 206 + `Content-Range: bytes 0-1048575/total` |
| 非法 Range 返回？ | 416 Range Not Satisfiable + Content-Range: bytes \*/total |
| 两种下载方式怎么选？ | 大文件下载用预签名 URL（减轻应用层压力），需要鉴权 / 断点续传 / 缓存协商用代理下载 |
| ETag 304 好处？ | 重复请求零数据传输，省带宽省时间 |

## 六、面试口述模板

> "下载链路有两种方式：**预签名 URL** 和 **代理下载**。预签名 URL 先查 Redis 缓存，命中直接返回，缓存 TTL 略小于签名有效期避免过期。代理下载支持 ETag 304（fileMd5 做 ETag，If-None-Match 匹配则零数据传输）和 Range 206（断点续传，视频播放器拖拽）。用 StreamingResponseBody 在 Tomcat 异步 I/O 线程中写出字节流，不阻塞 Servlet 线程。"
