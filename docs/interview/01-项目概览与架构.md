# 模块一：项目概览与架构

## 一、30秒项目介绍（开场必背）

> CloudChunk 是一个基于分片上传的云文件系统。用户传大文件时，前端把文件切成 5MB 的小块，并发往后端传；后端把每个小块存到 MinIO 对象存储，等所有分片都到齐了，服务端 Compose 合并成一个完整文件；合并完之后，再通过 RocketMQ 发异步消息，去做 MD5 校验和转码（图片压缩、视频转码、文档转 PDF）。技术栈是 Java 21 + Spring Boot 3.3 + MyBatis-Plus + Redis 7 + RocketMQ + MinIO + Caffeine。核心亮点有六个：秒传去重（引用计数）、断点续传（Redis+MySQL+MinIO 三层兜底）、去长事务（用 Little's Law 分析拆分）、Lua 原子进度、虚拟线程+Semaphore 背压、MQ 异步解耦。

**大白话解释每一层**：

- **客户端层**：用户操作的地方，浏览器、APP、小程序都算。用户选文件，前端 JS 把文件切成 5MB 的小块，一块一块传。
- **接入层**：Nginx 顶在最前面，负责限流、路由、转发。相当于一个"保安"，防止恶意请求打进来，也把请求分发到不同的后端机器。
- **应用层**：Spring Boot 集群，核心业务代码。分成几个服务：上传负责收分片，下载负责给文件，元数据负责管理文件信息，转码负责发 MQ 消息。StorageStrategy 是存储策略接口，MinIO/本地/OSS 三种实现，运行时切换。
- **消息层**：RocketMQ 是异步消息的"快递员"。文件合并完之后，告诉 MQ "我要转码"，MQ 把消息发给下游的 worker。主流程不阻塞，转码慢慢做，用户体验好。
- **异步处理层**：三个 worker（图片、视频、文档），各自消费 MQ 消息，干自己的活。这层可以独立部署、独立扩容，视频转码慢就多加机器，不影响主服务。
- **数据层**：MySQL 存文件元数据、引用计数、配额信息；Redis 存分片进度、秒传缓存、分布式锁。Redis Hash 天然适合存储 {chunkIndex: status}，查一个分片状态一条命令就搞定。
- **存储层**：真正存文件块的地方。MinIO 支持服务端 Compose 合并，这在面试里是个加分项，因为不用把文件块下载到应用内存再拼，省内存省网络。

## 三、模块依赖关系

```
cloudchunk-boot ──启动入口（打 war 包 / jar 包的入口）
  ├── cloudchunk-api ──Controller / Filter / 全局异常处理（对外接口）
  │     └── cloudchunk-core ──上传、下载、合并、校验、配额（核心业务）
  │           ├── cloudchunk-storage ──StorageStrategy 接口 + MinIO/Local/OSS 实现
  │           ├── cloudchunk-mq ──RocketMQ 生产者/消费者封装
  │           └── cloudchunk-infra ──Redis / MyBatis / 配置 / 工具
  └── cloudchunk-transcode ──图/视频/文档转码（独立可拆 Worker，可单独部署）
        ├── cloudchunk-storage
        ├── cloudchunk-mq
        └── cloudchunk-infra

所有模块 → cloudchunk-common ──工具/常量/枚举/异常（零依赖，谁都能用）
```

**关键规则**：

1. **单向依赖，禁止反向**：api 依赖 core，core 依赖 storage/mq/infra，反过来不行。一旦反向依赖，代码就变成一团乱麻了。
2. **transcode 与 core 同级，通过 MQ 解耦**：转码模块不直接依赖 core，它们通过 MQ 通信。核心流程只管发 MQ 消息，转码那边只管消费。这样转码模块可以独立部署、独立扩容，不影响主服务。
3. **common 零依赖**：所有模块都能用 common，但 common 自己不能依赖任何其他模块。它就是个"工具箱"，谁都能用，但不依赖谁。

## 四、技术选型理由（面试必背）

| 选型 | 理由 |
|------|------|
| Java 21 | 虚拟线程原生支持，I/O 密集型并发优势明显。上传分片是大量 I/O 等待的场景，虚拟线程让每个请求一个线程，线程数从几百撑到几万，吞吐量直接翻倍 |
| Spring Boot 3.3 | 生态成熟，Starter 机制降低集成成本。Spring Boot 3.3 原生支持虚拟线程，配置一下就开，不用自己折腾 |
| MyBatis-Plus | Lambda 查询 + 分页插件，开发效率高。比手写 SQL 少出错，比 JPA 更灵活，比原生 MyBatis 更省事 |
| MinIO | S3 兼容、支持 Compose 服务端合并、可私有化。Compose 合并意味着不用把分片下载到应用再拼，省内存省网络；私有化部署，数据不出机房 |
| RocketMQ | 事务消息 + TAG 路由 + 死信队列，国产化友好。TAG 路由让一个 Topic 下按图片/视频/文档分流，消费端各自消费；死信队列保证消息不会丢失；事务消息保证合并和转码消息的一致性 |
| Redis Hash | HSET/HGET 天然适合存储 {chunkIndex: status}，单命令原子。查一个分片状态一条 HGET 就搞定，不需要加锁，也不需要担心并发问题 |

## 五、面试必背

1. **30 秒介绍要能背出来**：CloudChunk 是什么、怎么做的、技术栈是什么、亮点在哪。这是面试开场白，背不出来后面就没法聊了。

2. **架构图要能画出来**：不用画得跟上面一样，但七层要能画出来：客户端→接入层→应用层→消息层→异步处理层→数据层→存储层。每层干啥要能说清楚。

3. **模块依赖关系要能说清楚**：boot → api → core → storage/mq/infra，transcode 与 core 同级通过 MQ 解耦，common 零依赖。三条规则：单向依赖、禁止反向、MQ 解耦。

4. **技术选型理由要能说清楚**：每个技术选型为什么用它，不用别的原因。面试官最爱问"为什么用 Redis 不用 Memcached"、"为什么用 MinIO 不用 FastDFS"。上面表格里背熟了就行。

5. **6 个核心亮点要能背出来**：
   - **秒传去重（引用计数）**：同样内容的文件，MD5 一致，直接引用计数 +1，不重复存文件。
   - **断点续传（Redis+MySQL+MinIO 三层兜底）**：Redis 缓存分片进度，MySQL 持久化记录，MinIO 持有文件块。任何一层挂了，其他层都能恢复进度。
   - **去长事务（Little's Law）**：用 Little's Law（L=λW）分析出，事务越长，系统并发瓶颈越厉害。把大事务拆成小步骤，异步处理，吞吐量直接提升。
   - **Lua 原子进度**：用 Redis Lua 脚本更新分片进度，原子操作，不会出现并发问题。
   - **虚拟线程+Semaphore 背压**：虚拟线程让每个请求一个线程，不浪费线程池；Semaphore 控制并发数，防止系统过载。
   - **MQ 异步解耦**：核心流程只管发消息，转码慢慢做，主服务不阻塞，用户体验好。
