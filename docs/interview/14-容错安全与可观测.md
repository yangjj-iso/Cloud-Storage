## 二、ResilientStorageDecorator — 装饰器重试

### 2.1 问题

MinIO 偶发网络抖动（TCP reset、DNS 超时），单次请求失败，但服务本身可用。如果不重试，用户就看到"上传失败"。

### 2.2 方案：装饰器模式 + 指数退避

```java
public class ResilientStorageDecorator implements StorageStrategy {
    private final StorageStrategy delegate;  // 被装饰的实际存储策略
    private final int maxRetries = 2;        // 最多重试 2 次
    private final long baseDelayMs = 100;    // 首次重试等 100ms

    private <T> T retryable(String operation, Supplier<T> action) {
        StorageException lastException = null;
        for (int attempt = 0; attempt <= maxRetries; attempt++) {
            try {
                return action.get();
            } catch (StorageException e) {
                lastException = e;
                if (attempt < maxRetries) {
                    long delay = baseDelayMs * (1L << attempt); // 指数退避：100ms, 200ms
                    Thread.sleep(delay);
                }
            }
        }
        throw lastException;
    }
}
```

### 2.3 注册方式

```java
// MinioAutoConfiguration.java
@Bean
public StorageStrategy minioStorageStrategy(MinioClient client, StorageProperties props) {
    MinioStorageStrategy raw = new MinioStorageStrategy(client, props);
    return new ResilientStorageDecorator(raw, 2, 100);  // 包一层装饰器
}
```

### 2.4 面试要点

| 问题 | 回答 |
|------|------|
| 为什么用装饰器不直接改 MinioStorageStrategy？ | 开闭原则，不改原有代码；可以按需组合（如加监控装饰器） |
| 为什么指数退避？ | 防止重试风暴——如果 100 个请求同时失败同时重试，MinIO 更扛不住 |
| PUT 不是非幂等吗？ | 我们的 PUT 用固定 objectKey，覆盖写，天然幂等 |
| 重试序列？ | 100ms → 200ms，总共最多 3 次尝试（首次 + 2 次重试） |

### 2.5 面试口述

> "MinIO 偶发超时用 ResilientStorageDecorator 装饰器处理。装饰器模式不改原有代码，符合开闭原则。2 次重试 + 指数退避（100ms, 200ms），防止重试风暴。PUT 用固定 key 天然幂等，可以安全重试。"

## 三、Watchdog 续期锁 — 防止长任务锁过期

### 3.1 问题

合并大文件时 `ComposeObject` 可能耗时数十秒。如果锁 TTL 是 5 分钟，正常情况够用。但如果 MinIO 响应慢、网络抖动，合并可能超过 5 分钟，锁过期了：

- 另一个请求获取到锁，触发**并发合并**
- 原持有者释放锁时 DEL 了**别人的锁**

### 3.2 方案：Watchdog 自动续期

```java
// RedisLock.java
public LockHandle tryLockWithWatchdog(String key, String token, Duration ttl) {
    if (!tryLock(key, token, ttl)) return null;

    // 每 TTL/3 续期一次（Redisson 默认策略：leaseTime=30s, renewInterval=10s）
    long renewIntervalMs = ttl.toMillis() / 3;
    ScheduledFuture<?> future = watchdogScheduler.scheduleAtFixedRate(() -> {
        // Lua 原子续期：仅当 token 匹配时才 PEXPIRE
        redisService.raw().execute(RENEW_SCRIPT, List.of(key), token, String.valueOf(ttl.toMillis()));
    }, renewIntervalMs, renewIntervalMs, TimeUnit.MILLISECONDS);

    return new LockHandle(key, token, future);
}

// 使用方式
var handle = redisLock.tryLockWithWatchdog(lockKey, token, Duration.ofMinutes(5));
try { return doMerge(s); }
finally { handle.unlock(); }  // 释放锁 + 停止续期
```

### 3.3 Lua 续期脚本

```lua
if redis.call('GET', KEYS[1]) == ARGV[1] then
    return redis.call('PEXPIRE', KEYS[1], ARGV[2])
else
    return 0  -- token 不匹配，说明锁已被别人拿了，不续期
end
```

### 3.4 面试要点

| 问题 | 回答 |
|------|------|
| 这就是 Redisson 的原理？ | 对，Redisson 的 watchdog 就是这个思路：后台线程每 leaseTime/3 续期 |
| 为什么 TTL/3？ | 留足余量，续期失败还有 2/3 的时间重试 |
| 续期时为什么要校验 token？ | 防止续了别人的锁——如果锁已经被别人拿了，不能续期 |
| daemon 线程？ | `setDaemon(true)`，JVM 关闭时不阻塞；`removeOnCancelPolicy` 取消后立即从队列移除 |

### 3.5 面试口述

> "合并大文件时用 Watchdog 续期锁。后台 daemon 线程每 TTL/3 执行 Lua 原子 PEXPIRE，仅当 token 匹配时才续期。业务完成后 handle.unlock() 停止续期 + 释放锁。这就是 Redisson watchdog 的核心原理。"

## 四、过期会话清理 — SessionCleanupScheduler

### 4.1 问题

用户上传中途放弃（关闭浏览器），分片对象残留在 MinIO。按 10MB/片 × 1000 片 = 10GB/会话，积累后存储成本巨大。

### 4.2 方案：定时任务 + 异步清理

```java
@Scheduled(fixedDelay = 3600_000, initialDelay = 60_000)  // 每小时执行
public void cleanExpiredSessions() {
    LocalDateTime deadline = LocalDateTime.now().minusMinutes(60); // 宽限 1 小时

    List<UploadSession> expired = sessionMapper.selectList(
            new LambdaQueryWrapper<UploadSession>()
                    .eq(UploadSession::getStatus, UploadSessionStatus.RUNNING)
                    .lt(UploadSession::getExpireAt, deadline)
                    .last("limit 100"));  // 每批最多 100 条

    for (UploadSession s : expired) {
        // CAS 更新：只更新 RUNNING 状态的（幂等）
        int affected = sessionMapper.update(null, new LambdaUpdateWrapper<UploadSession>()
                .eq(UploadSession::getFileId, s.getFileId())
                .eq(UploadSession::getStatus, UploadSessionStatus.RUNNING)
                .set(UploadSession::getStatus, UploadSessionStatus.FAILED));
        if (affected == 0) continue;  // 已被其他线程处理

        progressStore.clear(s.getFileId());
        cleanupPartsAsync(s);  // cleanupExecutor 异步删除 MinIO 分片
    }
}
```

### 4.3 两层保障

| 层次 | 机制 | 时效 |
|------|------|------|
| 应用层 | `SessionCleanupScheduler` 每小时扫描 | 过期后 1~2 小时清理 |
| 存储层 | MinIO Lifecycle Policy `parts/` 前缀 7 天过期 | 兜底，防止应用层遗漏 |

### 4.4 面试口述

> "过期会话用 @Scheduled 每小时扫描 status=RUNNING 且 expire_at < 1 小时前的会话，CAS 更新为 FAILED（幂等），然后用 cleanupExecutor 异步删除 MinIO 分片。MinIO Lifecycle Policy 7 天过期作为兜底。"

## 六、WebSocket 背压 — ConcurrentWebSocketSessionDecorator

### 6.1 问题

慢客户端（弱网、后台 Tab）阻塞 `sendMessage`，导致其他订阅者的进度推送延迟。

### 6.2 方案

```java
WebSocketSession decorated = new ConcurrentWebSocketSessionDecorator(
        session,
        5000,         // 发送超时 5 秒
        64 * 1024);   // 缓冲区上限 64KB
```

| 保护机制 | 行为 |
|----------|------|
| 发送超时 5s | 5s 内未发出 → 自动关闭连接 |
| 缓冲区溢出 64KB | 积压 > 64KB → 自动关闭连接 |
| 线程安全 | 多个分片并发完成不会消息交错 |
| 自动清理 | 发送失败的 session 立即从 registry 移除 |

### 6.3 面试口述

> "WebSocket 慢客户端用 ConcurrentWebSocketSessionDecorator 保护。设置 5s 发送超时 + 64KB 缓冲区上限，超限自动断开连接，不影响其他订阅者。这解决了 WebSocket 的慢消费者问题。"

### 7.1 新增指标

| `cloudchunk.executor.available{name=io/cleanup}` | Gauge | 执行器可用许可（越低越繁忙） |

### 7.2 StorageHealthIndicator

```java
@Component
public class StorageHealthIndicator implements HealthIndicator {
    @Override
    public Health health() {
        storageFactory.current().exists(bucket, "__health_check__");
        return Health.up().withDetail("latencyMs", elapsed).build();
    }
}
```

## 八、dedupChunks 批量优化

### 8.1 问题

原来 `dedupChunks` 逐个分片查 DB + 逐个 CopyObject，N 个分片就是 2N 次 RTT。

### 8.2 优化：批量查询 + 并行拷贝

```java
// 1. 批量 IN 查询（替代 N 次逐条 SELECT）
List<ChunkRecord> candidates = chunkMapper.selectList(
        new LambdaQueryWrapper<ChunkRecord>()
                .in(ChunkRecord::getChunkMd5, md5List)
                .eq(ChunkRecord::getStatus, ChunkStatus.DONE));

// 2. 并行拷贝（利用 ioExecutor 并发执行 MinIO CopyObject）
futures.add(CompletableFuture.supplyAsync(() -> {
    storage.copy(bucket, srcKey, bucket, dstKey);
    chunkMapper.upsert(rec);
    progress.markDone(fileId, idx, ttl);
    return idx;
}, ioExecutor));
```

### 8.3 性能对比

| 维度 | 优化前 | 优化后 |
|------|--------|--------|
| DB 查询 | N 次 SELECT | 1 次 IN 查询 |
| MinIO 拷贝 | 串行 N 次 RTT | 并行 ceil(N/64) 次 RTT |
| 总耗时（100 片） | ~100 × 50ms = 5s | ~1 × 5ms + 2 × 50ms = 105ms |

### 8.4 面试口述

> "dedupChunks 原来逐个查 DB + 逐个 CopyObject，N 片就是 2N 次 RTT。优化后：一次 IN 查询找到所有可复用分片，再用 ioExecutor 并行 CopyObject。100 片从 5 秒降到 100 毫秒。"
