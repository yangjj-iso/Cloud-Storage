# 模块十：限流（令牌桶）

## 一、问题是什么？为什么要限流？

想象一下这个场景：

- 有用户恶意疯狂上传文件，后端服务器扛不住，直接打挂了
- 有用户恶意疯狂下载文件，后端服务器也扛不住，又打挂了
- 所以我们必须限流：**每个用户每秒最多请求多少次**，不能让你无限刷

简单说，限流就是给每个用户加个"天花板"，防止你把服务器搞崩。

---

## 二、令牌桶 vs 漏桶 vs 滑动窗口（面试必背！）

这三种限流算法，面试必问，必须搞清楚：

| 算法 | 特点 | 通俗比喻 | 适用场景 |
|---|---|---|---|
| 令牌桶 | 允许突发（burst），匀速生成令牌 | 像收费站，桶里攒了令牌，你拿走就放行 | 分片上传（天然有突发需求） |
| 漏桶 | 平滑限流，匀速输出 | 像一个漏斗，水只能以固定速率流出 | 接口调用（平滑限流） |
| 滑动窗口 | 精确时间窗口统计 | 像一扇滑动窗口，看过去 1 分钟有多少次 | 秒杀活动（防刷） |

大白话解释：

- **令牌桶**：就像收费站有个桶，桶里会匀速放令牌（比如每秒 30 个）。你来了拿一个令牌就走，没令牌就等着。桶满了令牌就不再攒了。好处是：如果你之前没人用，桶里攒满了，你突然来了可以一口气拿走（突发）！
- **漏桶**：就像一个漏斗，水倒进去多少无所谓，但漏出来的速度是固定的，一滴一滴匀速出。你想要突发？不可能，都得匀速。
- **滑动窗口**：就像一扇窗户，看过去 1 分钟内你请求了多少次。精确到每秒，但算起来比较复杂。

---

## 三、为什么选令牌桶？

分片上传天然有突发需求：**前端并发上传 4-6 路分片，瞬间需要 4-6 个请求**。

- 令牌桶允许突发（burst），桶里攒满了令牌，你一口气来 6 个分片请求，我桶里够，6 个都放行
- 漏桶呢？它平滑限流，匀速输出，你 6 个分片同时请求，它只能按固定速率一个一个放，分片上传体验就差了
- 滑动窗口？精确但重，计算复杂

所以选令牌桶，**就是为分片上传的突发需求量身定制的**。

## 四、Redis Lua 令牌桶

核心代码在 `RateLimiter.java`：

```java
private static final RedisScript<Long> TOKEN_BUCKET = new DefaultRedisScript<>(
    "local rate = tonumber(ARGV[1]) " +
    "local cap  = tonumber(ARGV[2]) " +     // 桶容量
    "local now  = tonumber(ARGV[3]) " +
    "local d = redis.call('HMGET', KEYS[1], 'tk', 'ts') " +
    "local tokens = tonumber(d[1]) or cap " +   // 当前令牌数
    "local last   = tonumber(d[2]) or now " +    // 上次时间
    "local elapsed = math.max(0, now - last) " +
    "tokens = math.min(cap, tokens + elapsed * rate / 1000000) " +  // 令牌补充
    "local allowed = 0 " +
    "if tokens >= 1 then tokens = tokens - 1; allowed = 1 end " +  // 令牌扣1
    "redis.call('HMSET', KEYS[1], 'tk', tostring(tokens), 'ts', tostring(now)) " +
    "redis.call('EXPIRE', KEYS[1], math.ceil(cap / rate) + 1) " +
    "return allowed", Long.class);
```

用大白话逐步解释：

**第 1 步：HMGET 读取当前状态**
- 从 Redis Hash 里读出桶中剩余的令牌数（`tk`）和上次补充时间（`ts`）
- 如果是第一次（桶里没数据），令牌数默认等于桶容量（满了），时间等于当前时间

**第 2 步：计算应补充的令牌数**
- `elapsed = math.max(0, now - last)`，算距离上次过了多久
- `tokens = math.min(cap, tokens + elapsed * rate / 1000000)`，微秒精度！
- 举例：rate = 30（每秒 30 个令牌），elapsed = 0.5 秒 = 500000 微秒，那补充 500000 × 30 / 1000000 = 15 个令牌
- `math.min(cap, ...)`，别超过桶容量

**第 3 步：有令牌就扣1放行，否则拒绝**
- `if tokens >= 1 then tokens = tokens - 1; allowed = 1 end`
- 有令牌：扣1，放行，allowed = 1
- 没令牌：拒绝，allowed = 0

**第 4 步：HMSET 写回 + EXPIRE 自动过期**
- 写回新的令牌数和时间戳
- 设过期时间：`math.ceil(cap / rate) + 1`，桶满了之后，令牌最多能存多久，过期了自动清理

## 五、限流配置（per-user）

每个用户独立限流，互不干扰：

| 端点 | rate (tokens/s) | burst | 说明 |
|---|---|---|---|
| 上传分片 | 30 | 60 | 每秒 30 个分片，突发 60 个 |
| 下载 | 50 | 100 | 每秒 50 个下载，突发 100 个 |

大白话解释：

- **上传分片**：每秒稳稳地生成 30 个令牌，桶最多能攒 60 个。前端 4-6 路并发分片上传，完全够用
- **下载**：下载场景更频繁，每秒 50 个令牌，桶最多能攒 100 个

## 六、RateLimitFilter

在 Servlet Filter 层拦截，请求进来先过限流，超限直接返回 429 Too Many Requests。

```java
// 伪代码逻辑
if (rateLimiter.isAllowed(key, rate, burst)) {
    chain.doFilter(request, response);  // 放行
} else {
    response.setStatus(429);  // 429 Too Many Requests
    // 返回超限提示
}
```

为什么放在 Filter 层？因为**请求还没进到业务逻辑，就拦截掉了**，不浪费资源。

## 七、面试追问

| 问题 | 回答要点 |
|---|---|
| 令牌桶 vs 漏桶？ | 令牌桶允许突发（burst），适合分片上传；漏桶平滑限流，适合接口调用 |
| 为什么选令牌桶？ | 分片上传天然有突发需求，前端 4-6 路并发 |
| Lua 脚本在 Redis Cluster 兼容？ | 单 KEY 操作，天然同 slot，没问题 |
| 微秒精度？ | `elapsed * rate / 1000000`，用微秒级精度，令牌补充更精确 |
| per-user 限流？ | 令牌桶 Key = `cc:rate:upload:{userId}`，每个用户独立限流 |
| 限流超限返回？ | 429 Too Many Requests |
