# 模块十二：存储策略模式

## 一、为什么需要存储策略模式？

用大白话解释：

- **开发时**用本地文件系统，不需要启动 MinIO
- **私有化部署**用 MinIO（S3 兼容、可集群）
- **公有云部署**用阿里云 OSS
- 如果不抽象，业务代码里到处都是 if-else，切换存储后端要改业务代码
- **用策略模式：业务代码只调 `StorageStrategy` 接口，不关心底层实现**
- 一行配置切换，业务代码零改动

---

## 二、StorageStrategy 接口

核心代码：

```java
public interface StorageStrategy {
    String type();  // minio / local / oss
    PutResult put(PutRequest request);
    InputStream get(GetRequest request);
    RangeStream getRange(GetRangeRequest request);
    ComposeResult compose(ComposeRequest request);
    String presignDownload(String bucket, String objectKey, Duration ttl);
    void delete(String bucket, String objectKey);
    void deleteBatch(String bucket, List<String> keys);
    boolean exists(String bucket, String objectKey);
    ObjectStat stat(String bucket, String objectKey);
}
```

**一句话总结：** 不管你底层用的是 MinIO、本地文件系统还是阿里云 OSS，对上层业务来说，就是调这 9 个方法——上传、下载、范围读、合并、预签名、删除、批量删除、判断存在、查元信息。接口统一，底层随便换。

## 三、工厂与配置

```java
@Component
public class StorageStrategyFactory {
    private final Map<String, StorageStrategy> strategies;  // Spring 自动注入所有实现

    public StorageStrategyFactory(List<StorageStrategy> all,
                                  @Value("${cloudchunk.storage.type}") String type) {
        this.strategies = all.stream()
            .collect(toMap(StorageStrategy::type, identity()));
        this.currentType = type;
    }

    public StorageStrategy current() { return byType(currentType); }
}
```

**大白话解释：** Spring 把所有 `StorageStrategy` 实现都注入进来，工厂按 `type()` 字符串分好类。你要哪个实现，调 `current()` 就行——由配置文件决定。

配置文件：

```yaml
cloudchunk:
  storage:
    type: minio    # minio | local | oss
    default-bucket: cloudchunk
```

**一句话总结：** 改一行配置 `type: minio` → `type: oss`，整个项目从 MinIO 切到阿里云 OSS，业务代码一行都不用改。

---

## 四、三种实现

| 实现 | type | 适用场景 | 说明 |
|---|---|---|---|
| `MinioStorageStrategy` | `minio` | 私有化部署 | S3 兼容、支持 Compose 合并、可集群 |
| `LocalStorageStrategy` | `local` | 开发调试 | 本地文件系统，用 FileChannel.transferTo 合并 |
| `AliyunOssStorageStrategy` | `oss` | 公有云部署 | 阿里云 OSS，用 UploadPartCopy 合并 |

**大白话解释：**

- **开发环境**：本机跑，不用装 MinIO，`type: local`，直接用本地磁盘，省事。
- **私有化部署**：客户机房，`type: minio`，S3 兼容，能集群，数据不丢。
- **公有云部署**：阿里云，`type: oss`，直接用阿里云的对象存储，稳定可靠。

## 五、双写灰度（迁移场景）

```java
@Component
@ConditionalOnProperty("cloudchunk.storage.migrate.enabled")
public class DualWriteStorageStrategy implements StorageStrategy {
    public PutResult put(PutRequest req) {
        PutResult r = primary.put(req);       // 写新存储
        migrateExecutor.execute(() -> secondary.put(req));  // 异步双写旧存储
        return r;
    }
    public InputStream get(GetRequest req) {
        try { return primary.get(req); }
        catch (NotFoundException e) { return secondary.get(req); }  // 回源
    }
}
```

**大白话解释：**

- **写**：新存储先写，旧存储异步双写（不阻塞主流程）
- **读**：新存储先读，命中返回；未命中则回源到旧存储（读降级）

**为什么要这样？** 举个例子：你要从 MinIO 迁移到阿里云 OSS。迁移期间，新文件写 OSS，但旧文件可能还在 MinIO。所以：

- **写**：新存储（OSS）先写，旧存储（MinIO）异步双写——保证不丢数据，新存储也不阻塞主流程。
- **读**：新存储（OSS）先读，命中就直接返回；没命中，说明这文件还在旧存储（MinIO）里，回源读一下。

**一句话总结：** 灰度迁移的核心思路：**写新存储先写，异步双写旧存储；读新存储先读，命中返回，未命中回源到旧存储。**

## 六、本地 Compose 实现

用 `FileChannel.transferTo` 实现零拷贝：

```java
@Override
public ComposeResult compose(ComposeRequest req) {
    Path target = resolve(req.bucket(), req.targetKey());
    try (FileChannel out = FileChannel.open(target, CREATE, TRUNCATE_EXISTING, WRITE)) {
        for (String srcKey : req.sourceKeys()) {
            try (FileChannel in = FileChannel.open(src, READ)) {
                long size = in.size();
                long transferred = 0;
                while (transferred < size) {
                    transferred += in.transferTo(transferred, size - transferred, out);
                }
            }
        }
    }
}
```

**大白话解释：**

- **Compose 合并**：把多个分块文件合并成一个完整文件。大文件上传分块存，最后合并成完整文件。
- **零拷贝**：`transferTo` 底层走操作系统层面的 `sendfile`，数据不经过应用层内存（直接从磁盘到磁盘），省内存、省带宽。
- **为什么需要循环？** `transferTo` 不保证一次传输完所有数据，所以需要循环确保传输完成。

## 八、面试口述模板

> "存储策略模式：接口 + 工厂 + 三种实现。`StorageStrategy` 接口定义统一 API，MinIO/Local/OSS 三种实现。工厂通过配置切换，业务代码零改动。双写灰度：迁移场景写新存储先写，异步双写旧存储；读新存储先读，命中返回，未命中回源到旧存储。"

---

**记住这几句话：**

1. **接口统一**：`StorageStrategy` 定义 9 个方法，业务代码只调接口。
2. **工厂切换**：`StorageStrategyFactory` 按配置选实现，一行配置切换。
3. **三种实现**：MinIO（私有化）、Local（开发）、OSS（公有云）。
4. **双写灰度**：写新存储先写，异步双写旧存储；读新存储先读，命中返回，未命中回源到旧存储。
5. **零拷贝合并**：`FileChannel.transferTo`，数据不走应用层内存，效率高。
