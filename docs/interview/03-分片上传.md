# 模块三：分片上传（核心必考！）

## 一、分片上传是什么？为什么要做？

先说大文件直接上传的痛点：

- 你上传一个 1GB 的视频，**网络一断，全白费**，从头再来
- 1GB 文件直接灌到服务器，**服务器内存撑不住**，一两个大文件就 OOM
- HTTP 请求本身就有超时限制，**传着传着就超时了**，用户只能干瞪眼

分片上传就是解决这些问题的：

**把大文件切成 5MB 的小块，一块一块上传，失败了就重传这一块。**

举个最直观的例子：你搬家，是把所有家具一股脑塞进一辆大卡车，还是分装到好几辆小车上？小车装得快，哪辆坏了换哪辆，不影响其他车。分片上传就是"分装"思路。

**分片上传的好处**：

| 好处 | 大白话解释 |
|------|-----------|
| 并发上传 | 5 块分片同时上传，5 条路一起走，速度翻倍 |
| 断点续传 | 传到第 3 块断了，第 1、2 块不用重传，接着传第 3 块 |
| 失败重传 | 哪块失败就重传哪块，不用整块重来 |
| 服务端合并 | 分片都传完了，后端把分片拼回完整文件 |

## 二、分片上传的流程

### 前端分片算法（大白话版）

前端拿到一个文件，怎么切片？三步：

1. **算分片大小**：`chunkSize = 5MB`（默认，可配 1~100MB）
2. **算分片总数**：`chunkTotal = ceil(fileSize / chunkSize)`，比如 1GB 文件 = 1024MB / 5MB ≈ 205 个分片
3. **循环切片**：`for i in [0, chunkTotal)`，每个分片切出来 → 算 MD5 → POST 上传

用伪代码表示：

```
chunkSize = 5MB
chunkTotal = ceil(fileSize / chunkSize)

for i in [0, chunkTotal):
    start = i * chunkSize
    end = min(start + chunkSize, fileSize)
    chunk = file.slice(start, end)          // 切出这一块
    chunkMd5 = md5(chunk)                    // 算这一块的 MD5
    POST /upload/chunk, { fileId, chunkIndex: i, chunkMd5, chunk }  // 上传
```

**并发控制**：前端用 Promise Pool 限制并发 4~6 路，不会同时 200 个请求一起打出去，把服务器冲垮。

### 分片命名规范

分片在 MinIO 里怎么存？有统一规则：

```
{bucket}/upload/{yyyyMMdd}/{fileId}/part.{chunkIndex:06d}
```

举个例子：

```
mybucket/upload/20250615/abc123/part.000000
mybucket/upload/20250615/abc123/part.000001
mybucket/upload/20250615/abc123/part.000002
...
```

- `chunkIndex:06d`：6 位数字补零，第 1 个分片是 `part.000000`，第 205 个是 `part.000204`
- 这样排列整齐，合并时按顺序读取拼接就行

## 三、核心代码：uploadChunk()（项目最核心的方法！）

这是 `UploadService.uploadChunk()` 的源码，逐行讲清楚：

```java
public ChunkUploadResponse uploadChunk(String fileId, int chunkIndex, String chunkMd5,
                                       long chunkSize, InputStream data) throws IOException {
    UploadSession s = requireRunningSession(fileId);

    // 幂等：重复上传同一分片直接返回
    if (progress.isDone(fileId, chunkIndex)) {
        ChunkRecord exist = chunkMapper.selectOne(...);
        return new ChunkUploadResponse(fileId, chunkIndex, exist.getEtag(), 1,
                progress.doneCount(fileId) == s.getChunkTotal());
    }

    // 流式 MD5：DigestInputStream 边传边算，零堆分配
    StorageStrategy storage = storageFactory.current();
    String partKey = ObjectKeyUtils.partKey(fileId, chunkIndex);
    MessageDigest digest = MessageDigest.getInstance("MD5");
    DigestInputStream dis = new DigestInputStream(data, digest);
    PutResult pr = storage.put(PutRequest.of(s.getBucket(), partKey, dis, chunkSize, MimeUtils.OCTET_STREAM));

    // MD5 校验失败 → 异步删除错误分片
    String actualMd5 = Md5Utils.hex(digest.digest());
    if (!actualMd5.equalsIgnoreCase(chunkMd5)) {
        ioExecutor.execute(() -> {   // BoundedVirtualThreadExecutor
            try { storage.delete(s.getBucket(), partKey); } catch (Exception ignored) {}
        });
        throw BizException.of(ErrorCode.CHUNK_MD5_MISMATCH);
    }

    // MySQL 幂等 upsert（INSERT ON DUPLICATE KEY UPDATE）
    ChunkRecord rec = new ChunkRecord();
    rec.setFileId(fileId); rec.setChunkIndex(chunkIndex);
    rec.setChunkMd5(actualMd5); rec.setChunkSize((int) chunkSize);
    rec.setEtag(pr.etag()); rec.setStatus(ChunkStatus.DONE);
    chunkMapper.upsert(rec);

    // Redis Lua 原子进度：HSETNX + HINCRBY + EXPIRE 一次 RTT
    int done = progress.markDone(fileId, chunkIndex, properties.getChunk().getSessionTtl());
    return new ChunkUploadResponse(fileId, chunkIndex, pr.etag(), 1, done == s.getChunkTotal());
}
```

### 逐行解释

**1️⃣ 幂等检查：重复上传同一分片直接返回**

```java
if (progress.isDone(fileId, chunkIndex)) {
    ChunkRecord exist = chunkMapper.selectOne(...);
    return new ChunkUploadResponse(fileId, chunkIndex, exist.getEtag(), 1,
            progress.doneCount(fileId) == s.getChunkTotal());
}
```

大白话：前端网络抖了，重传了同一个分片，后端已经存过了，**直接返回成功，不重复处理**。这就是幂等性——同一操作做多次结果一样。

**2️⃣ 流式 MD5：DigestInputStream 边传边算，不额外 buffer 整块数据**

```java
MessageDigest digest = MessageDigest.getInstance("MD5");
DigestInputStream dis = new DigestInputStream(data, digest);
```

大白话：常规做法是先把 5MB 全读出来，再算 MD5，再传到 MinIO——那 5MB 数据就要在内存里待两遍。`DigestInputStream` 是**边传边算 MD5**，数据流过一遍就完事了，内存零额外开销。

**3️⃣ MinIO PUT：把分片写到 MinIO 对象存储**

```java
PutResult pr = storage.put(PutRequest.of(s.getBucket(), partKey, dis, chunkSize, MimeUtils.OCTET_STREAM));
```

大白话：把分片流式写入 MinIO，`partKey` 就是按命名规范生成的路径，比如 `mybucket/upload/20250615/abc123/part.000003`。

**4️⃣ MD5 校验：失败则异步删除错误分片**

```java
String actualMd5 = Md5Utils.hex(digest.digest());
if (!actualMd5.equalsIgnoreCase(chunkMd5)) {
    ioExecutor.execute(() -> {   // BoundedVirtualThreadExecutor
        try { storage.delete(s.getBucket(), partKey); } catch (Exception ignored) {}
    });
    throw BizException.of(ErrorCode.CHUNK_MD5_MISMATCH);
}
```

大白话：传上来的 MD5 和前端传的 MD5 不一致？说明数据坏了，必须**异步删除**这个分片，然后抛出 `CHUNK_MD5_MISMATCH` 异常让前端重传。异步删除是怕删除操作本身也有异常，用虚线程池（BoundedVirtualThreadExecutor）不阻塞主流程。

**5️⃣ MySQL 幂等 upsert：INSERT ON DUPLICATE KEY UPDATE**

```java
chunkMapper.upsert(rec);
```

大白话：往 MySQL 里写一条分片记录。如果主键冲突了（同一个 fileId + chunkIndex），就 UPDATE 而不是 INSERT，**不会重复插入**，保证幂等性。

**6️⃣ Redis Lua 原子进度：HSETNX + HINCRBY + EXPIRE**

```java
int done = progress.markDone(fileId, chunkIndex, properties.getChunk().getSessionTtl());
```

大白话：在 Redis 里标记这个分片已上传，用 Lua 脚本把三步合成一步（HSETNX + HINCRBY + EXPIRE），**一次 RTT 搞定**，保证原子性。返回的 `done` 是已上传完成的分片数，如果 `done == chunkTotal`，说明所有分片传完了，可以合并。

## 四、去 @Transactional（必考！）

### 为什么去掉 @Transactional？

**这是面试官最爱问的考点，一定要理解清楚！**

先看问题：如果 `uploadChunk()` 方法加了 `@Transactional`，会发生什么？

**MinIO PUT 耗时约 200ms**（网络 I/O），这 200ms 期间，DB 连接被占用着，因为事务还没结束，连接不能释放。

HikariCP 默认连接池大小是 30，100 并发时：

```
占用连接数 = QPS × 平均响应时间
           = 100 × 0.2s（MinIO PUT 200ms）
           = 20 连接
```

20 个连接占了 30 个池子的 2/3！稍有波动就连接池耗尽，**全站雪崩**——所有需要 DB 连接的接口都卡住。

### Little's Law 推导（面试加分项）

**Little's Law**（排队论基本定律）：**L = λ × W**

- L = 同时需要的连接数
- λ = 请求到达率（QPS）
- W = 平均响应时间

**加了 @Transactional**：

```
L = 100 × 0.2s = 20 连接 ← 接近池上限（30），危险！
```

**去掉 @Transactional**：

```
L = 100 × 0.005s = 0.5 连接 ← 连接池几乎无压力！
```

去掉后 DB 操作只占 5ms，MinIO PUT 的 200ms 不再占用 DB 连接，连接池从 20 降到 0.5，利用率从 67% 降到 2%！

### 去掉后怎么保证一致性？

去掉 `@Transactional` 不是说不用事务了，而是**两步 DB 写都是幂等原子操作**，不需要事务保证：

| 操作 | 原子性保证 | 原理 |
|------|-----------|------|
| `chunkMapper.upsert()` | MySQL 原子 | INSERT ON DUPLICATE KEY UPDATE 是一条 SQL，天然原子 |
| `progress.markDone()` | Redis 原子 | Lua 脚本在 Redis 中单线程执行，天然原子 |

两步都是幂等的：即使中间某步失败，重试也不会产生脏数据。

## 五、两种上传端点

Controller 层有两个上传端点，对应两种不同的请求方式：

### multipart/form-data：兼容模式

```
POST /upload/chunk
Content-Type: multipart/form-data

fileId: abc123
chunkIndex: 3
chunkMd5: xxxxxxx
file: (分片数据)
```

大白话：**multipart 就像你先把货物搬到临时仓库，再从临时仓库搬到 MinIO。**

Tomcat 收到 multipart 请求后，会先把分片数据写到临时文件（磁盘），再让你从临时文件中读取。多了一步磁盘读写，多了一次 I/O，但**兼容性最好**，所有浏览器都支持。

### application/octet-stream：高性能模式

```
POST /upload/chunk/abc123/3
Content-Type: application/octet-stream
X-Chunk-Md5: xxxxxxx

(分片数据直接作为 body)
```

大白话：**octet-stream 就像直接从卡车上搬进 MinIO，省了一步！**

`request.getInputStream()` 直接拿到分片数据流，**零临时文件**，直传 MinIO。数据从网络到 MinIO，流式传输，内存零额外开销。

### 对比

| 维度 | multipart/form-data | application/octet-stream |
|------|-------------------|------------------------|
| 临时文件 | 有（Tomcat 写磁盘） | 无（直传 MinIO） |
| 磁盘 I/O | 多一次 | 零额外 |
| 兼容性 | 所有浏览器都支持 | 旧浏览器可能不支持 |
| 性能 | 稍慢 | 更快 |
| 适用场景 | 通用兼容 | 高性能要求 |

## 六、面试追问

| 问题 | 回答要点 |
|------|---------|
| 为什么去 @Transactional？ | MinIO PUT ~200ms，包在事务里会持有 DB 连接，Little's Law = QPS x 耗时 = 100 x 0.2s = 20 连接，接近池上限，去掉后 DB 5ms，100 x 0.005 = 0.5 连接 |
| 去掉后怎么保证一致性？ | 两步 DB 写都是幂等原子操作：upsert（MySQL 原子）+ Lua 脚本（Redis 原子） |
| 分片 MD5 校验怎么做？ | DigestInputStream 边传边算，零额外 buffer，流式传输时顺便算 MD5 |
| 分片大小为什么选 5MB？ | MinIO Compose 最小分片约束 + 过小会放大元数据开销（分片太多，MySQL/Redis 记录太多） |
| 两种上传端点区别？ | multipart 有临时文件（Tomcat 先写磁盘再读），octet-stream 零临时文件直传 MinIO |
| 分片 MD5 校验失败后怎么办？ | 异步删除错误分片（用虚线程池），抛 CHUNK_MD5_MISMATCH 让前端重传 |
| 分片上传的幂等性怎么保证？ | Redis 进度 + MySQL upsert，重复上传同一分片直接返回成功 |
| 前端并发怎么控制？ | Promise Pool 限制 4~6 路并发，不会同时打出去 200 个请求 |
