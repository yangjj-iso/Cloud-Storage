# 模块六：服务端合并（Compose）

## 一、合并是什么？为什么要做？

分片上传完后，分片是散在 MinIO 里的，就像一袋子里装了一堆碎片，用户需要的是一个完整的文件，不是一堆碎片。

**合并就是：把所有分片拼接成一个完整文件。**

我们用的是 **MinIO Compose Object 服务端拼接**，数据不经应用层——文件直接在 MinIO 服务器里拼接，不经过我们的 Java 应用。

**好处：**
- 不占应用层带宽——文件数据不用从 MinIO 下载到 Java 再上传回去
- 不占用应用层内存——大文件在 Java 里拼，内存直接爆了

一句话总结：**让 MinIO 自己干活，Java 应用只当指挥官。**

---

## 二、合并的触发时机

合并有两种触发方式：

| 触发方式 | 说明 | 场景 |
|---|---|---|
| **隐式触发** | 最后一个分片上传成功后，判断已完成数 == 总数，则自动触发合并 | 正常流程，用户不用手动操作 |
| **显式触发** | 前端调 `POST /upload/merge/{fileId}`（幂等） | 网络抖动、隐式触发失败后，前端主动重试 |

显式触发是**幂等的**——意思是：合并成功了，你再调一次，不会报错，也不会重复合并。

## 三、合并的核心代码：事务与 I/O 分离

合并的核心代码分两部分，**事务外**和**事务内**，这是设计的关键。

```java
// UploadService.doMerge() —— 合并编排（事务外！）
private MergeResult doMerge(UploadSession s) {
    // ① 校验分片是否全部到齐
    mergeTx.markMerging(s.getFileId());   // 状态 → MERGING

    // ② MinIO Compose（纯 I/O，在事务外！）
    StorageStrategy storage = storageFactory.current();
    List<String> sources = new ArrayList<>(s.getChunkTotal());
    for (int i = 0; i < s.getChunkTotal(); i++) {
        sources.add(ObjectKeyUtils.partKey(s.getFileId(), i));
    }
    ComposeResult cr = storage.compose(new ComposeRequest(s.getBucket(), s.getObjectKey(), sources));

    // ③ 事务内写库（短事务 ~5ms）
    mergeTx.finalizeSuccess(s, storage.type());

    // ④ 投递校验消息 + 异步清理分片
    checksumProducer.publish(msg);
    progress.clear(s.getFileId());
    cleanupPartsAsync(s.getBucket(), sources);
    return new MergeResult(...);
}
```

```java
// MergeTransactionService.java（独立 @Service，让 @Transactional 代理生效）
@Transactional(rollbackFor = Exception.class)
public void finalizeSuccess(UploadSession s, String storageType) {
    FileMeta meta = new FileMeta();
    meta.setFileId(s.getFileId());
    meta.setStatus(FileStatus.MERGED);            // 非 AVAILABLE！
    fileMetaService.insert(meta);
    sessionMapper.update(...);  // session status=COMPLETED
    quotaService.addUsed(s.getOwnerId(), s.getFileSize());
}
```

**用大白话讲这四步：**

1. **先校验分片是否全部到齐**——人齐了才能开会，分片齐了才能合并
2. **MinIO Compose 拼合**——这是纯 I/O 操作，在事务外面！因为 MinIO 拼文件可能要几秒，放在事务里会把数据库连接占着，时间一长就完蛋了
3. **短事务写库**——拼合成功后，事务内插入 `file_meta`、更新 `session`、扣配额。这三步都是写库操作，放在一起约 5ms，事务很短，不会锁太久
4. **投递校验消息 + 异步清理分片**——拼合完了，发个校验任务，异步删掉分片释放空间

**为什么要事务外做 I/O？** 因为 MinIO Compose 拼合可能要几秒，数据库事务撑不住，连接会被占着，时间一长就超时了。把 I/O 和事务分开，事务只做几毫秒的事，稳稳的。

## 四、为什么 MergeTransactionService 要独立成 @Service？

先看代码，`MergeTransactionService` 是独立的一个 `@Service`，不是 `UploadService` 的内部方法。

**为什么要独立出来？**

**因为 Spring AOP 代理不拦截自调用！**

啥意思？看这个：

```
UploadService.doMerge() 调 this.finalizeSuccess()
    → this 是目标对象，不是代理对象
    → @Transactional 不生效！
    → 事务没开，数据库操作要么全成功，要么全失败，没保证
```

独立出来之后：

```
UploadService.doMerge() 调 mergeTx.finalizeSuccess()
    → mergeTx 是注入的代理对象
    → @Transactional 生效！
    → 事务开了，保证原子性
```

**一句话：自己调自己方法，Spring 的代理不拦截，事务注解等于摆设。独立成另一个 Service，注入后调用，代理才能生效。**

## 五、为什么状态是 MERGED 不是 AVAILABLE？

合并完成后，状态是 `MERGED`，不是 `AVAILABLE`。用户下载文件时，只能下载 `AVAILABLE` 状态的文件。

**两阶段设计：**

```
Compose 后 MERGED → 异步校验 MD5 通过后才 AVAILABLE
```

**为什么不直接 AVAILABLE？**

- 如果直接 AVAILABLE，校验 MD5 发现文件坏了，用户可能已经下载了坏文件
- 先 MERGED，校验通过后才 AVAILABLE，用户下载不到坏文件
- 这就像工厂质检——先生产出来，再质检，合格了才上架

**一句话：先 MERGED，校验通过后才 AVAILABLE，用户下载不到坏文件。**

## 六、MinIO 分批 Compose

MinIO Compose 有一些限制：

| 限制 | 说明 |
|---|---|
| 单次最多 10,000 个源对象 | 一次最多拼 10000 个分片 |
| 每个源对象最小 5 MB（最后一个除外） | 分片太小不行，MinIO 不支持 |
| **这就是为什么前端选 5MB 切片** | 5MB 是 MinIO 的下限，低于这个就拼不了 |

**超限怎么办？分批 Compose！**

```
N 个分片 → 批次1: part.0~999 合为 temp.0 →
            批次2: part.1000~1999 合为 temp.1 →
            ... → 再合并 temp.0..K 为最终对象
```

先合成中间对象，再二次合并为最终对象。就像拼乐高——先拼几个大块，再把大块拼成最终作品。

## 七、合并锁

合并前用 Redis 分布式锁防止并发：

```
Redis SETNX cc:upload:merge-lock:{fileId} EX 300
```

- **SETNX**：只有锁不存在时才设置成功，防止并发
- **EX 300**：5 分钟超时自动释放，防止死锁

**为什么需要锁？** 最后一个分片上传成功后，前端可能同时触发隐式和显式合并，两个请求同时进来，不锁就合并两次。

**一句话：Redis SETNX + 5min TTL，防止并发合并。**

## 八、面试追问

| 问题 | 回答要点 |
|---|---|
| 为什么独立成 @Service？ | Spring AOP 代理不拦截自调用，UploadService 调自身方法 @Transactional 不生效 |
| 状态为什么是 MERGED 不是 AVAILABLE？ | 两阶段：Compose 后 MERGED → 异步校验 MD5 通过后才 AVAILABLE，防止用户下载坏文件 |
| Compose 失败怎么办？ | 不进 finalizeSuccess 事务，file_meta 未插入，前端可重试 merge |
| 合并锁怎么实现？ | Redis SETNX + TTL 5min，防止并发合并 |
| MinIO Compose 限制？ | 单次 10000 对象，每个最小 5MB，超限分批合并 |
| MinIO Compose 合并的好处？ | 数据不经应用层，不占应用层带宽和内存 |
