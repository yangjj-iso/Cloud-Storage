# 模块二：秒传（Instant Upload）

## 一、秒传是什么？为什么要做？

用大白话讲，秒传就是：**用户上传一个文件，如果这个文件已经有人上传过了，就不需要再上传了，直接返回下载链接。**

举个例子：你网盘里存了《西游记》的 PDF，你室友也想上传同一份《西游记》，系统发现 MD5 一样，判定"文件已经存在了"，直接秒传——不用再传一遍，体验就像"秒级完成"一样。

**好处：**

- **节省 35% 的存储空间**：同一个文件只存一份，不重复落盘
- **用户体验秒级完成**：上传瞬间完成，不用等进度条
- **原理：MD5 是文件的唯一标识**，相同文件 MD5 一定相同，所以用 MD5 判断文件是否已存在

## 二、秒传是怎么做的？

核心代码来自 `UploadService.init()` 方法，下面用伪代码逐步讲解：

```java
public InitUploadResponse init(InitUploadRequest req, long userId) {
    validateInit(req);
    quotaService.checkCapacityOrThrow(userId, req.getFileSize());

    // 分布式锁：防止同一 MD5 并发创建多个会话
    String lockKey = RedisKeys.uploadInstantLock(req.getFileMd5());
    String token = IdUtils.uuid32();
    if (!redisLock.tryLock(lockKey, token, Duration.ofMinutes(10))) {
        throw BizException.of(ErrorCode.UPLOAD_IN_PROGRESS, req.getFileMd5());
    }
    try {
        // ① 秒传：MD5 命中已有 AVAILABLE 文件 → ref_count+1，直接返回下载 URL
        var instant = fileMetaService.findAvailableByMd5(req.getFileMd5());
        if (instant.isPresent()) {
            FileMeta m = instant.get();
            fileMetaService.incRefCount(m.getFileId());
            String url = storageFactory.current()
                    .presignDownload(m.getBucket(), m.getObjectKey(), properties.getChunk().getSessionTtl());
            return InitUploadResponse.instant(m.getFileId(), url);
        }

        // ② 续传：存在 RUNNING 会话 → 返回已上传列表 + 缺失列表
        UploadSession existing = sessionMapper.selectOne(new LambdaQueryWrapper<UploadSession>()
                .eq(UploadSession::getFileMd5, req.getFileMd5())
                .eq(UploadSession::getStatus, UploadSessionStatus.RUNNING)
                .last("limit 1"));
        if (existing != null && existing.getExpireAt().isAfter(LocalDateTime.now())) {
            List<Integer> uploaded = loadOrRebuildUploaded(existing);
            List<Integer> missing = buildMissing(existing.getChunkTotal(), uploaded);
            return InitUploadResponse.resume(existing.getFileId(), existing.getChunkSize(),
                    existing.getChunkTotal(), uploaded, missing, existing.getExpireAt());
        }

        // ③ 新建会话
        UploadSession s = createSession(req, userId);
        progress.init(s.getFileId(), s.getChunkTotal(), properties.getChunk().getSessionTtl());
        return InitUploadResponse.upload(s.getFileId(), s.getChunkSize(), s.getChunkTotal(), s.getExpireAt());
    } finally {
        redisLock.unlock(lockKey, token);
    }
}
```

用大白话逐步解释三步走：

### 第①步：秒传——查 file_meta 表，命中就直接返回

```sql
SELECT * FROM file_meta WHERE file_md5 = ? AND status = 'AVAILABLE' LIMIT 1
```

- 如果查到了，说明这个文件已经有人上传过了，存储里已经有了
- **不需要再上传，直接引用计数 +1（ref_count + 1）**
- 生成预签名下载链接，直接返回给前端
- 用户看到的是"上传瞬间完成"，这就是"秒传"

### 第②步：续传——查 upload_session 表，有正在进行的会话

- 如果 file_meta 表里没命中，但 upload_session 表里有一条 **RUNNING 状态的会话**还没过期
- 说明之前有人上传过同一文件，但还没传完（传了一半断了）
- 这种情况下，返回"已上传列表 + 缺失列表"，用户可以**续传**，不用重新来

### 第③步：新建——都没有，就创建新会话

- file_meta 表也没命中，upload_session 表也没有正在进行的会话
- 那就创建一条新的 upload_session 记录
- Redis Hash 初始化分片进度（每个 chunk 是 0，还没传）
- 返回会话信息，前端开始真正上传分片

## 三、秒传的幂等控制（分布式锁）

### 为什么需要分布式锁？

场景：**用户同时从两个 Tab 上传同一文件**，两次 `/upload/init` 并发请求

不加锁会发生什么？

1. 两个请求都去查 file_meta 表，发现"不存在"
2. 两个请求都创建新的 upload_session
3. 最终上传完成合并时，会往 file_meta 表里插入两条记录
4. **同一个文件存了两份，存储空间浪费，数据也冗余**

### 方案：Redis SETNX 排它锁

```java
String lockKey = RedisKeys.uploadInstantLock(req.getFileMd5());
String token = IdUtils.uuid32();
if (!redisLock.tryLock(lockKey, token, Duration.ofMinutes(10))) {
    throw BizException.of(ErrorCode.UPLOAD_IN_PROGRESS, req.getFileMd5());
}
```

本质就是执行：

```
SETNX cc:upload:lock:{md5} token EX 600
```

- **锁 key**：`cc:upload:lock:{md5}`，同一个 MD5 只有一把锁
- **超时**：10 分钟，防止异常情况锁不释放
- **释放**：初始化结束（不管成功失败）立即 `DEL`，释放锁
- **异常兜底**：如果服务挂了，锁没释放，10 分钟后自动过期

**为什么用 MD5 而不是文件名做锁 key？** 因为同一文件可能改名，但 MD5 不变，所以要用 MD5 作为锁粒度。

## 四、引用计数（ref_count）

### 秒传命中后：ref_count + 1

秒传命中后，**不复制对象**，只做引用计数 +1。这样：

- 用户 A 之前上传了《西游记.pdf》，file_meta 表里 ref_count=1
- 用户 B 再上传同一文件，秒传命中，ref_count=2
- 用户 C 也有，秒传命中，ref_count=3
- **存储里只存了一份《西游记.pdf》，但三个人都"看到"了这个文件**

### 删除文件时：ref_count - 1

- 用户 A 删除了，ref_count=3-1=2，**不删对象**，因为还有别人在用
- 用户 B 删除了，ref_count=2-1=1，**不删对象**，还有一个人在用
- 用户 C 删除了，ref_count=1-1=0，**这时候才真正删除存储对象**

### 效果

**这样同一个文件只存一份，就能节省约 35% 的存储空间**。这也是很多网盘产品（百度网盘、阿里云盘）都在用的核心方案。

## 五、面试追问

| 问题 | 回答要点 |
|---|---|
| 秒传怎么做的？ | `WHERE file_md5=? AND status=AVAILABLE LIMIT 1`，命中后 `ref_count+1`，不复制对象 |
| 并发秒传怎么办？ | Redis SETNX 排它锁，10min 超时，保证只有一个创建成功 |
| 配额检查为什么在最前面？ | 快速失败，避免无效的存储操作 |
| MD5 前端算的是什么？ | 前端用 spark-md5 增量计算整文件 MD5，大文件分块读取 |
| 大文件 MD5 算得慢？ | 文件 > 1GB 时，退化为分片 MD5 级联（牺牲秒传命中率换取客户端性能） |
