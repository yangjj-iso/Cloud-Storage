# 模块十一：虚拟线程 + Semaphore 背压

## 一、虚拟线程是什么？为什么用 Java 21？

用大白话解释：

- 传统线程（平台线程）很重，一个线程占 1MB 栈，1000 个线程就占 1GB
- 虚拟线程（Virtual Thread）很轻，几乎不占内存，可以创建百万个
- I/O 密集型场景（上传/下载/Redis 操作）用虚拟线程，线程在等待 I/O 时不阻塞平台线程
- Java 21 原生支持虚拟线程，Spring Boot 3.3 开启 `spring.threads.virtual.enabled=true`

## 二、为什么虚拟线程需要 Semaphore 限制？

面试必考！用大白话解释：

- 虚拟线程虽然创建开销极小，但下游系统（MinIO 连接数、Redis 连接数）有物理上限
- 如果 10 万并发直接涌向 MinIO/Redis，MinIO 连接池会耗尽
- Semaphore 保护的是下游资源而非线程本身
- 比喻：虚拟线程就像可以无限增加的工人，但 MinIO 连接就像只有 64 个工位，工人再多也不能超过工位数

## 三、BoundedVirtualThreadExecutor

核心代码：

```java
public class BoundedVirtualThreadExecutor implements Executor, AutoCloseable {
    private final Semaphore permits;
    private final ExecutorService delegate;

    public BoundedVirtualThreadExecutor(String name, int maxConcurrent, long acquireTimeoutMs) {
        this.permits = new Semaphore(maxConcurrent);
        this.delegate = Executors.newThreadPerTaskExecutor(
                Thread.ofVirtual().name("bvte-" + name + "-", 0).factory());
    }

    @Override
    public void execute(Runnable task) {
        boolean acquired = permits.tryAcquire(acquireTimeoutMs, TimeUnit.MILLISECONDS);
        if (!acquired) throw new RejectedExecutionException("bounded-executor full");
        delegate.execute(() -> {
            try { task.run(); }
            finally { permits.release(); }
        });
    }
}
```

用大白话逐步解释：

1. **Semaphore(permits)**：限制最大并发数，比如 64 个 permits，最多同时跑 64 个任务
2. **tryAcquire(timeout)**：获取许可，失败则抛 `RejectedExecutionException`——超限了就拒绝，不让任务挤进来
3. **delegate.execute(() -> { task.run() })**：在虚拟线程中执行任务，拿到许可才能干活
4. **finally { permits.release() }**：不管任务成功还是失败，执行完后释放许可，让下一个任务进来

## 四、两个 Executor 配置

| Executor | permits | 说明 |
|---|---|---|
| ioExecutor | 64 | MinIO PUT / MQ 补偿 |
| cleanupExecutor | 32 | 合并后分片清理 |

- ioExecutor 64：MinIO 的连接池最多 64 个，并发上传也最多 64 个
- cleanupExecutor 32：清理分片不需要那么快，32 个并发够了

## 五、MinIO OkHttp Client 调优

默认 OkHttp: ConnectionPool(5) + maxRequestsPerHost(5)，高并发下成为吞吐瓶颈。

自定义：

- ConnectionPool(256, 5min)
- Dispatcher.maxRequests = 256, maxRequestsPerHost = 256
- connectTimeout = 10s, readTimeout = 60s, writeTimeout = 60s

大白话解释：

- 默认只有 5 个连接、5 个并发请求，100 个上传排队就炸了
- 调到 256 个连接 + 256 个并发，吞吐量拉满
- 超时时间也要适当延长，上传大文件 60s 读写超时比较合理

## 六、Tomcat 配置

```yaml
server.tomcat:
  threads.max: 400
  max-connections: 10000
  accept-count: 500
  connection-timeout: 20s
```

开启 `spring.threads.virtual.enabled=true`，实际请求由虚拟线程承载，平台线程只做 NIO 事件分发。

大白话解释：

- Tomcat 的平台线程只做"接活"，接到请求后丢给虚拟线程干活
- 平台线程只负责 NIO 事件分发，不阻塞在 I/O 上
- 400 个线程可以应对 10000 个连接，排队 500 个，够用了

## 七、面试追问

| 问题 | 回答要点 |
|---|---|
| 虚拟线程为何加 Semaphore？ | 保护下游资源非线程本身，虚拟线程轻量但 MinIO/Redis 连接池有限 |
| 虚拟线程 vs 平台线程？ | 虚拟线程轻量（~KB），可创建百万个；平台线程重（~1MB），最多几百个 |
| 虚拟线程适用场景？ | I/O 密集型（上传/下载/Redis 操作），CPU 密集型不适合 |
| Semaphore 限流 vs RejectedExecutionException？ | 限流：Semaphore 限制并发；超限：抛 RejectedExecutionException |
| OkHttp Client 调优？ | ConnectionPool(256) + maxRequestsPerHost(256)，默认 5 太小 |
