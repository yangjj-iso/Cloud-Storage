# 模块九：缓存体系

## 一、问题是什么？为什么要做缓存？

用大白话讲：

- 文件元数据（`file_meta`）是高频读数据——每次查文件列表、查文件详情都要查 MySQL
- 如果每次都查 MySQL，QPS 1000 的话 MySQL 顶不住
- 用缓存：Caffeine L1（本地缓存）+ Redis L2（分布式缓存）两级缓存

**一句话总结：** 读得多、写得少的数据，用缓存挡住大部分请求，让 MySQL 不被拖垮。

---

## 二、两级缓存设计

### 2.1 L1：Caffeine 本地缓存

核心代码在 `FileMetaService.java`：

```java
// L1: Caffeine 本地缓存
this.localCache = Caffeine.newBuilder()
        .maximumSize(c.getFileMetaMaxSize())
        .expireAfterWrite(c.getFileMetaTtl())
        .recordStats()
        .build();

// 读：L1 → DB
public Optional<FileMeta> findById(String fileId) {
    if (localCache != null) {
        FileMeta cached = localCache.getIfPresent(fileId);
        if (cached != null) return Optional.of(cached);
    }
    FileMeta m = mapper.selectOne(...);
    if (m != null && localCache != null) localCache.put(fileId, m);
    return Optional.ofNullable(m);
}

// 写穿透：任何写操作都清除 L1 + L2
private void invalidateCache(String fileId) {
    if (localCache != null) localCache.invalidate(fileId);
    redis.delete(RedisKeys.fileMeta(fileId));
    redis.delete(RedisKeys.fileUrl(fileId));
}
```

**大白话解释：**

- `maximumSize`：本地缓存最多存多少条
- `expireAfterWrite`：写入后多久过期（TTL）
- `recordStats`：开启统计，后面可以看命中率
- `findById`：先查 Caffeine，命中直接返回；没命中查 MySQL，查完写回 Caffeine
- `invalidateCache`：写操作时，Caffeine 和 Redis 都要清掉，保证下次读到的数据是最新的

### 2.2 L2：Redis 缓存

Redis 里存两个 key：

| Redis Key | 内容 | 说明 |
|---|---|---|
| `cc:file:meta:{fileId}` | 文件元数据 Hash | 文件名、大小、状态等 |
| `cc:file:url:{fileId}` | 预签名 URL 缓存 | 避免频繁生成预签名 URL |

### 2.3 读取流程

```
1. 先查 Caffeine 本地缓存 → 命中（~100ns）→ 直接返回
2. 未命中 → 查 Redis → 命中（~1ms）→ 写入 Caffeine → 返回
3. Redis 也未命中 → 查 MySQL → 写入 Redis + Caffeine → 返回
```

**大白话：** 读完之后，把数据往回填——MySQL 查到后写 Redis，Redis 查到后写 Caffeine。下次再读，就能从更快的缓存里拿到。

### 2.4 写穿透

```
所有写操作（updateStatus、updateExtra、delete）都调用 invalidateCache：
1. 清除 Caffeine 本地缓存
2. 清除 Redis 缓存
3. TTL 兜底最终一致
```

**大白话：** 只要数据变了，就把缓存里的旧数据清掉。下次读的时候自然会从 MySQL 拿到最新数据，再回填缓存。没有清不干净的，因为 TTL 兜底。

## 三、性能对比

| 缓存 | 延迟 | 说明 |
|---|---|---|
| Caffeine L1 | ~100ns | 本地缓存，零网络开销 |
| Redis L2 | ~1ms | 网络往返一次 |
| MySQL | ~5ms | 网络往返 + 磁盘IO |

**一句话：** Caffeine L1 减少 99% 的 Redis/DB 访问。

**算笔账：** 假设 QPS 1000，Caffeine 命中率 99%，那每秒只有 10 次请求打到 Redis/MySQL，轻松扛住。

## 四、缓存一致性保证

### 4.1 写穿透

所有写操作在 `invalidateCache` 中同时清除 Caffeine + Redis + TTL 兜底。

**大白话：** 写完数据库，立刻把缓存清掉。这叫"写穿透"——写操作直接穿透缓存，让缓存失效。

### 4.2 多实例部署

多实例部署时 Caffeine 本地缓存不一致怎么办？

- Redis Pub/Sub 广播失效事件：实例 A 写入后广播失效 → 实例 B/C 收到后清除本地 Caffeine
- TTL 兜底最终一致

**大白话：** 比如部署了 3 个实例 A、B、C。实例 A 的 Caffeine 被清了，但 B、C 的 Caffeine 还是旧数据。怎么办？实例 A 写完之后，通过 Redis Pub/Sub 广播一条消息，告诉 B、C "这个 fileId 的缓存失效了，赶紧清掉！"

### 4.3 为什么要用 TTL 兜底？

- 万一 Pub/Sub 消息丢失（Redis 宕机、网络抖动）
- TTL 兜底保证最终一致（最坏情况 5 分钟后过期）

**大白话：** Pub/Sub 广播虽然快，但不是 100% 可靠。万一 Redis 宕机了、网络抖动了，消息丢了，B、C 的本地缓存就一直是脏的。TTL 兜底就是"最后一道保险"——最坏情况下，5 分钟后过期，下次读到的就是新数据。

## 五、Caffeine 可观测

Caffeine 开启 `recordStats()`，绑定 Micrometer：

| 指标 | 含义 |
|---|---|
| `cache.gets{cache=fileMeta, result=hit/miss}` | 命中/未命中次数 |
| `cache.size{cache=fileMeta}` | 当前缓存条目数 |
| `cache.evictions{cache=fileMeta}` | 驱逐次数 |

通过 `/actuator/metrics` 查看。

**大白话：** 开了 `recordStats()` 就能知道命中率多少、缓存里存了多少条、驱逐了多少条。上线后盯着命中率，如果命中率低于 90%，说明缓存大小或 TTL 要调。

## 六、面试追问

| 问题 | 回答要点 |
|---|---|
| 为什么不用 Redis 做一级缓存？ | 本地 Caffeine ~100ns vs Redis ~1ms，高频读场景下 L1 减少 99% 的 Redis/DB 访问 |
| Caffeine 缓存一致性怎么保证？ | 写穿透：所有写操作在 invalidateCache 中同时清除 Caffeine + Redis。TTL 兜底。 |
| 多实例部署怎么保证？ | Redis Pub/Sub 广播失效事件，各实例收到后清除本地 Caffeine |
| TTL 兜底为什么？ | 万一 Pub/Sub 消息丢失，TTL 兜底保证最终一致 |
| 为什么要用 Caffeine 不用 Guava？ | Caffeine 性能更好（W-TinyLFU 算法），Spring Boot 3.x 官方推荐 |
