## 三条主线

### 上传主线

`秒传 -> 分片上传 -> 断点续传 -> Redis Lua 进度 -> 服务端合并 -> 异步校验`

对应文档：02、03、04、05、06、07。

### 下载主线

`元数据查询 -> 预签名 URL -> 流式下载 -> ETag / Range -> 缓存失效`

对应文档：08、09。

### 高并发主线

`去长事务 -> 裸流上传 -> Redis Lua -> 虚拟线程 + Semaphore -> 令牌桶限流 -> 指标观测`

对应文档：03、05、10、11。

## 面试口述建议

- 先讲业务价值：大文件稳定上传、断点续传、秒传、异步转码。
- 再讲核心难点：分片进度一致性、服务端合并、下游存储保护。
- 最后讲可量化优化：减少 Redis RTT、避免长事务、Range 下载、缓存命中、背压限流。

一句话版本：

> CloudChunk 是一个分布式文件存储服务，我重点解决了大文件上传的稳定性和性能问题：前端分片并计算 MD5，后端用 Redis Lua 原子记录进度，通过 MinIO Compose 做服务端合并，再用 RocketMQ 异步校验和转码；同时用虚拟线程配合 Semaphore 做背压，避免高并发下把 MinIO、Redis 或数据库打满。
