# 模块四：断点续传

## 一、断点续传是什么？为什么要做？

先说个场景：

你上传一个 2GB 的文件，传到 80% 的时候，网络突然断了，或者你不小心把浏览器关了，或者电脑重启了。

如果没有断点续传 → 你只能从头再传，心态炸裂。

有了断点续传 → 重新打开后，之前传过的分片还在，只要接着传剩下的 20% 就行。

**大白话总结：**
- 用户的网络断了、浏览器关了、电脑重启了，上传进度还在
- 重新打开后，不用重新上传，只要传还没传完的就行
- 好处：用户不用重新上传，大文件上传体验好

---

## 二、断点续传的触发条件

什么时候会触发断点续传？三种情况：

1. **浏览器刷新 / 关闭后重新打开** → 最常见的情况
2. **网络中断后恢复** → 网络断了又连上
3. **显式点击"继续上传"按钮** → 用户主动操作

前端怎么做的？前端用 **IndexedDB / LocalStorage** 把 `fileId` 持久化存下来，用户重新打开页面后，前端拿着这个 `fileId` 调 `POST /upload/init`，后端一看："这个 fileId 我之前见过，而且还有分片没传完"，就判断是续传模式，返回已上传的分片列表，前端接着传剩下的就行。

---

## 三、三层兜底：Redis → MySQL → MinIO

这是整个断点续传的核心逻辑，面试必问！

先看核心代码：

```java
private List<Integer> loadOrRebuildUploaded(UploadSession s) {
    List<Integer> uploaded = progress.uploaded(s.getFileId());  // ① 先查 Redis
    if (!uploaded.isEmpty()) return uploaded;

    // ② Redis 为空 → 从 MySQL 查 DONE 的 chunk_record
    Set<Integer> db = new TreeSet<>();
    chunkMapper.selectList(new LambdaQueryWrapper<ChunkRecord>()
            .eq(ChunkRecord::getFileId, s.getFileId())
            .eq(ChunkRecord::getStatus, ChunkStatus.DONE))
            .forEach(r -> db.add(r.getChunkIndex()));
    if (db.isEmpty()) return List.of();

    // ③ 再与 MinIO 实际存在的 part 对象取交集
    Set<Integer> confirmed = new TreeSet<>(db);
    var list = storageFactory.current().list(s.getBucket(),
            ObjectKeyUtils.partPrefix(s.getFileId()), 20000);
    Set<Integer> store = new TreeSet<>();
    list.forEach(it -> {
        int idx = ObjectKeyUtils.parsePartIndex(it.objectKey());
        if (idx >= 0) store.add(idx);
    });
    confirmed.retainAll(store);  // 关键：交集

    // ④ 重建 Redis 进度
    progress.rebuild(s.getFileId(), s.getChunkTotal(), confirmed, properties.getChunk().getSessionTtl());
    return new ArrayList<>(confirmed);
}
```

用大白话逐层拆解：

### ① 第一层：Redis Hash（正常情况）

- Redis 里存着每个分片的上传进度（Hash 结构，key = fileId，value = 已上传的分片索引）
- 如果 Redis 有数据，直接返回，**最简单、最快速**
- **这是正常情况**，大部分时候 Redis 都在，数据都在

### ② 第二层：MySQL（Redis 丢了的情况）

- Redis 丢了的情况有哪些？比如：TTL 过期了、Redis 重启了、数据被淘汰了
- 这时候从 MySQL 查 `chunk_record` 表，状态是 `DONE` 的记录
- 能拿到哪些分片已经写入了数据库记录

### ③ 第三层：MySQL + MinIO 取交集（为什么取交集？）

这是**最关键的一步**，面试必问！

为什么要取交集？因为 MySQL 和 MinIO **可能出现不一致**：

- **MySQL 落了但 MinIO PUT 可能失败** → MySQL 记了分片完成，但 MinIO 那边其实没存上
- **MinIO PUT 成了但 MySQL 写入失败** → MinIO 存了分片文件，但 MySQL 记录没写成功

所以：**两边都有，才算真的完成！** 取交集就是最保险的方案。

### ④ 重建 Redis 进度

用交集确认后的数据，把 Redis 进度重建回来，下次再查就快了。

---

**总结一句话：**

```
Redis 里有 → 直接用
Redis 丢了 → MySQL 查记录
MySQL 也不够 → MinIO 实际文件 + MySQL 取交集，两边都有才算真的完成
最后重建 Redis 进度
```

## 四、过期处理

上传会话不可能一直留着，过期了得清理。

### 过期机制

- `upload_session.expire_at` 默认 24 小时（可配置）
- 定时任务每小时扫描：`status=0 AND expire_at < NOW()`

### 清理步骤（四步）

1. **置 status=4（过期）** → 标记会话已过期
2. **清理 MinIO 中 upload/\*/fileId/\* 前缀对象** → 删除分片文件
3. **删除 Redis 进度 Key** → 清理缓存
4. **删除 chunk_record** → 清理数据库记录

大白话：24 小时内没传完的，过期了就把 MinIO 的分片、Redis 的进度、MySQL 的记录都删了，干净利落。

## 五、面试追问

| 问题 | 回答要点 |
|---|---|
| 断点续传怎么做的？ | Redis 进度 + MySQL + MinIO 交集，三层兜底重建 |
| Redis 进度丢失怎么办？ | `loadOrRebuildUploaded()`：MySQL chunk_record + MinIO list 取交集重建 |
| 为什么要取 MySQL 和 MinIO 的交集？ | MySQL 落了但 MinIO PUT 可能失败，或反过来，两边都有才算真的完成 |
| 过期处理？ | 24h TTL + 定时任务清理过期会话和 MinIO 分片 |
| 上传会话过期时间？ | 24h，可配置 |
